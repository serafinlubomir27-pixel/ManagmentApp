Slovenský jazyk

Táto bakalárska práca sa zaoberá návrhom a implementáciou desktopovej aplikácie pre projektový manažment s názvom ManagmentApp. Aplikácia je vyvinutá v programovacom jazyku Python s využitím knižníc CustomTkinter pre grafické rozhranie, SQLite3 pre lokálnu databázu, matplotlib pre vizualizácie a reportlab pre export do PDF. Jadrom systému je implementácia metódy kritickej cesty (CPM — Critical Path Method), ktorá umožňuje výpočet ES, EF, LS, LF hodnôt, celkovej rezervy (float) a identifikáciu kritickej cesty projektu. Aplikácia ďalej obsahuje Gantt chart, sieťový diagram závislostí, správu tímu s trojúrovňovou hierarchiou, kalendárny pohľad, systém komentárov a príloh k úlohám. Osobitná pozornosť bola venovaná návrhu duálneho dizajnového systému, ktorý podporuje prepínanie medzi tmavým a svetlým režimom prostredníctvom dvojíc farebných tokenov (light, dark) natívne podporovaných v CustomTkinter. V nadväznosti na vydanie verzie 1.0 bola realizovaná séria rozšírení (v1.1): automatizovaná sada 64 unit testov (pytest), live vyhľadávanie úloh, scrollovateľný Gantt chart, notifikácie blížiacich sa termínov, rekurzívne zobrazenie hierarchie tímu a CSV import/export. Verzia 1.2 zaviedla duálny databázový backend umožňujúci transparentné prepínanie medzi lokálnym SQLite a cloudovým PostgreSQL (Supabase) bez akejkoľvek zmeny repozitárnej alebo prezentačnej vrstvy.

Kľúčové slová: projektový manažment, metóda kritickej cesty, CPM, Python, CustomTkinter, desktopová aplikácia, SQLite, PostgreSQL, Supabase, Gantt chart, unit testovanie, design systém

English

This bachelor thesis presents the design and implementation of a desktop project management application called ManagmentApp. The application is developed in Python using the CustomTkinter library for the graphical user interface, SQLite3 for local data storage, matplotlib for visualizations, and reportlab for PDF export. The core of the system is an implementation of the Critical Path Method (CPM), which computes ES, EF, LS, LF values, total float, and identifies the critical path of a project. Additional features include a Gantt chart, CPM network diagram, team management with a three-level hierarchy, calendar view, and a comment and attachment system for tasks. Special attention was paid to the design of a dual-mode design system that supports toggling between dark and light themes using native CustomTkinter dual-value color tokens. Following the v1.0 release, a series of enhancements were delivered in v1.1: an automated suite of 64 unit tests (pytest), live task search, a scrollable Gantt chart, deadline notifications, recursive team hierarchy tree, and CSV import/export. Version 1.2 introduced a dual database backend enabling transparent switching between local SQLite and cloud PostgreSQL (Supabase) without any changes to the repository or presentation layers.

Keywords: project management, critical path method, CPM, Python, CustomTkinter, desktop application, SQLite, PostgreSQL, Supabase, Gantt chart, unit testing, design system

1. Úvod

1.1 Motivácia a kontext problému

Projektový manažment patrí medzi kľúčové disciplíny moderného podnikania. Nezávisle od odvetvia — či ide o vývoj softvéru, stavebníctvo, konzultačné služby alebo freelancing — každý projekt pozostáva zo série vzájomne závislých úloh, ktoré musia byť splnené v stanovenom čase a v rámci dostupných zdrojov. Nesprávne plánovanie alebo ignorovanie kritických závislostí medzi úlohami vedie k oneskoreniam, prekročeniu rozpočtu a neúspešnému dokončeniu projektov.

Napriek existencii početných nástrojov na projektový manažment — od jednoduchých tabuliek v Exceli po komplexné SaaS platformy ako Asana, Jira alebo Monday.com — mnohé z nich nesplňujú potreby malých tímov a freelancerov z niekoľkých dôvodov: sú príliš komplexné pre každodenné použitie, vyžadujú trvalé internetové pripojenie, ukladajú citlivé dáta na vzdialených serveroch, alebo sú finančne nedostupné pre jednotlivcov a malé firmy.

Táto práca reaguje na potrebu jednoduchej, lokálnej a bezplatnej desktopovej aplikácie pre projektový manažment, ktorá pritom implementuje matematicky fundované metódy plánovania. Konkrétne ide o implementáciu metódy kritickej cesty (CPM — Critical Path Method), ktorá identifikuje najdlhšiu sekvenciu vzájomne závislých úloh v projekte — tzv. kritickú cestu. Akékoľvek oneskorenie na tejto ceste automaticky predĺži celý projekt.

Aplikácia ManagmentApp vznikla postupnou evolúciou: pôvodný prototyp bol navrhnutý ako webová aplikácia na platforme Lovable (PathFlow), následne prepísaný do Pythonu a dokončený ako plnohodnotná desktopová aplikácia s dôrazom na moderný dizajn, robustnosť a rozšíriteľnosť.

1.2 Ciele práce

Hlavným cieľom tejto bakalárskej práce je navrhnúť a implementovať funkčnú desktopovú aplikáciu pre projektový manažment, ktorá:

  1. Implementuje algoritmus CPM s výpočtom ES, EF, LS, LF hodnôt, celkovej rezervy a identifikáciou kritickej cesty,
  2. Vizualizuje projektový plán prostredníctvom Ganttovho diagramu a sieťového CPM diagramu,
  3. Podporuje správu tímu s trojúrovňovou hierarchiou (admin – manažér – zamestnanec),
  4. Umožňuje export projektového reportu do PDF vrátane grafu a tabuľky úloh,
  5. Implementuje moderný duálny dizajnový systém s podporou tmavého aj svetlého režimu,
  6. Je navrhnutá s ohľadom na budúcu migráciu na cloudové riešenie (Repository Pattern).

2. Analýza existujúcich riešení a teoretické základy

2.1 Metóda kritickej cesty (CPM)

Metóda kritickej cesty (Critical Path Method — CPM) je technika sieťovej analýzy vyvinutá v 50. rokoch 20. storočia spoločne firmami DuPont a Remington Rand pre plánovanie priemyselných projektov [1]. Metóda pracuje s orientovaným acyklickým grafom (DAG), kde uzly reprezentujú úlohy a hrany závislosti medzi nimi.

2.1.1 Základné pojmy

SymbolAnglický názovSlovenský prekladDefinícia
ESEarliest StartNajskorší začiatokNajskorší čas, kedy môže úloha začať
EFEarliest FinishNajskorší koniecES + trvanie úlohy
LSLatest StartNajneskorší začiatokNajneskorší čas začatia bez predĺženia projektu
LFLatest FinishNajneskorší koniecNajneskorší čas ukončenia bez predĺženia projektu
TFTotal FloatCelková rezervaLS − ES (koľko sa dá oneskoriť bez dopadu)
Tabuľka 2.1 — Základné CPM parametre

Algoritmus prebieha v dvoch prechodoch cez sieť. Dopredný prechod (Forward Pass) počíta ES a EF pre každú úlohu postupom od začiatku siete ku koncu. Spätný prechod (Backward Pass) počíta LF a LS smerom od konca ku začiatku. Úlohy s nulovou celkovou rezervou (TF = 0) ležia na kritickej ceste — akékoľvek ich oneskorenie predĺži celý projekt.

2.1.2 Ilustratívny príklad

Uvažujme jednoduchý projekt so štyrmi úlohami a nasledovnými závislosťami: A(3d) → B(2d), A(3d) → C(4d), B a C → D(1d). Algoritmus CPM vypočíta nasledovné hodnoty:

ÚlohaTrvanieESEFLSLFTFKritická
A303030✓
B235572—
C437370✓
D178780✓
Tabuľka 2.2 — Výsledky CPM pre ilustratívny príklad. Kritická cesta: A → C → D = 8 dní.

Úloha B má celkovú rezervu 2 dni — môže sa začať o 2 dni neskôr bez dopadu na celkové trvanie projektu. Úlohy A, C, D ležia na kritickej ceste a majú nulovú rezervu.

2.2 Ganttov diagram

Ganttov diagram je horizontálny pruhový graf, ktorý vizualizuje projektový harmonogram. Každá úloha je reprezentovaná pruhom, ktorého ľavý okraj zodpovedá ES a dĺžka trvania úlohy. Henry Gantt navrhol túto vizualizáciu okolo roku 1910 [2], pričom dnes patrí k najrozšírenejším nástrojom projektového manažmentu.

V kombinácii s CPM algoritmus Ganttov diagram obohacuje o ďalší rozmer: svetlomodrý polotransparentný pruh za každou úlohou zobrazuje celkovú rezervu (TF), čo okamžite vizuálne komunikuje, ktoré úlohy majú časový priestor a ktoré nie.

3. Analýza požiadaviek

3.1 Cieľové skupiny a používateľské roly

Aplikácia je primárne určená pre tri cieľové skupiny: freelancerov a solopodnikateľov, ktorí spravujú viacero projektov súčasne a potrebujú prehľadný nástroj bez zbytočnej komplexity; malé tímy (5–20 ľudí) kde je hierarchia plochá a komunikácia priama; a manažérov s podriadenými, ktorí potrebujú priraďovať úlohy a sledovať vyťaženie tímu.

Systém implementuje trojúrovňovú hierarchiu rolí:

  • Admin — má prístup ku všetkým funkciám, vidí všetkých používateľov, môže pridávať nových členov tímu,
  • Manager — spravuje projekty a priraďuje úlohy členom svojho tímu (priamym podriadeným),
  • Employee — zobrazuje priradené úlohy, aktualizuje ich stav, pridáva komentáre a prílohy.

4. Návrh systému

4.1 Architektúra aplikácie

Aplikácia je navrhnutá ako trojvrstvová architekúra (Three-tier architecture) so striktnou separáciou zodpovedností. Každá vrstva komunikuje výhradne s susednými vrstvami — prezentačná vrstva nikdy neposiela SQL priamo do databázy.

Prezentačná vrstva (UI)
ui/screens/*.py, ui/components/*.py
CustomTkinter widgety, matplotlib canvasy
↕
Business logika
logic/cpm_engine.py, logic/cpm_manager.py
logic/task_manager.py, logic/export_manager.py
↕
Repository vrstva
repositories/task_repo.py, user_repo.py
repositories/project_repo.py, activity_repo.py
↕
Databáza
SQLite3 — my_project_app.db

Kľúčovým architektonickým rozhodnutím je implementácia Repository Pattern. Celá databázová logika je zapuzdrená v module repositories/ — UI nikdy nevykonáva SQL priamo. Toto rozhodnutie bolo prijaté s výhľadom na budúcu migráciu na cloudovú databázu (PostgreSQL/Supabase), kde stačí vymeniť repozitárnu vrstvu bez akejkoľvek zmeny v UI alebo business logike.

# Príklad toku dát od UI po databázu
UI → logic.task_manager.add_task(project_id, name, user_id)
        → task_repo.create_task(project_id, name, assigned_to, created_by)
              → SQLite: INSERT INTO tasks (...) VALUES (?)

Tento prístup zaručuje spätú kompatibilitu s existujúcimi databázovými súbormi.

4.4 Duálny dizajnový systém

CustomTkinter natívne podporuje dvojice farebných hodnôt vo formáte (light_value, dark_value). Pri volaní ctk.set_appearance_mode("light") alebo ("dark") sa automaticky prepne na príslušnú hodnotu v každom widgete. Táto vlastnosť bola využitá pri návrhu centralizovaného dizajnového systému v module ui/theme.py.

# Ukážka duálnych farebných tokenov v ui/theme.py
BG_MAIN    = ("#F5F7FF", "#0A0E1A")   # pozadie stránky
BG_CARD    = ("#FFFFFF",  "#101629")   # karty / elevované plochy
PRIMARY    = ("#4B7FFF", "#5A8FFF")   # primárna brand farba
BORDER     = ("#E8E8ED", "#1E263A")   # orámovanie kariet
SUCCESS    = ("#1DB86D", "#1DB86D")   # rovnaká v oboch módoch

# Font factory — lazy init (CTkFont musí čakať na CTk inicializáciu)
_font_cache: dict = {}

def get_font(size: int = FONT_SIZE_BASE, weight: str = "normal") -> ctk.CTkFont:
    key = (size, weight)
    if key not in _font_cache:
        _font_cache[key] = ctk.CTkFont(family="Segoe UI", size=size, weight=weight)
    return _font_cache[key]

Pre matplotlib grafy, ktoré nepoužívajú CTk systém, bola implementovaná pomocná funkcia mode_color(token), ktorá číta aktuálny mód pri čase renderovania:

def mode_color(token: tuple[str, str]) -> str:
    """Vráti správnu farbu podľa aktuálneho appearance mode."""
    return token[1] if ctk.get_appearance_mode() == "Dark" else token[0]

5.4 Business logika — CPM Engine

5.4.1 Dátové štruktúry

CPM engine používa dve dátové triedy implementované ako Python dataclasses:

from dataclasses import dataclass, field

@dataclass
class CPMTask:
    id: int;  name: str;  duration: int
    dependencies: list[int] = field(default_factory=list)
    delay_days: int = 0;  status: str = "pending"
    # Vypočítané polia (Forward + Backward Pass):
    es: int = 0;  ef: int = 0;  ls: int = 0;  lf: int = 0
    total_float: int = 0;  is_critical: bool = False

@dataclass
class CPMResult:
    tasks: list[CPMTask]
    critical_path: list[int]              # IDs kritických úloh
    project_duration: int                 # celkové trvanie
    project_duration_without_delays: int  # bez oneskorení
    total_project_delay: int
    is_valid: bool;  errors: list[str]

5.4.2 Implementácia algoritmu

Algoritmus prebieha v šiestich krokoch. Prvým krokom je detekcia cyklov pomocou DFS (hĺbkového prehľadávania). Každý uzol má jeden z troch stavov: BIELE (nenavštívený), SIVÉ (v aktuálnej ceste), ČIERNE (spracovaný). Ak DFS narazí na sivý uzol, existuje cyklus a CPM nie je možné vypočítať — algoritmus vráti CPMResult(is_valid=False).

def detect_cycle(tasks: list[CPMTask]) -> bool:
    task_map = {t.id: t for t in tasks}
    WHITE, GRAY, BLACK = 0, 1, 2
    color = {t.id: WHITE for t in tasks}

    def dfs(node_id):
        color[node_id] = GRAY
        for dep in task_map[node_id].dependencies:
            if dep not in task_map: continue
            if color[dep] == GRAY: return True  # cyklus!
            if color[dep] == WHITE and dfs(dep): return True
        color[node_id] = BLACK
        return False

    return any(dfs(t.id) for t in tasks if color[t.id] == WHITE)

Druhým krokom je topologické triedenie algoritmom Kahn. Výsledné poradie garantuje, že pred spracovaním každej úlohy sú spracované všetky jej závislosti.

Tretím krokom je Forward Pass — iterácia cez topologicky usporiadané úlohy. ES každej úlohy je maximum EF všetkých predchodcov plus delay_days:

es = max(ef[dep] for dep in dependencies) + delay_days  # (alebo 0 + delay)
ef = es + duration

Štvrtým krokom je Backward Pass — iterácia v obrátenom topologickom poradí. LF poslednej úlohy je rovné celkovému trvaniu projektu. Pre ostatné úlohy je LF minimum LS všetkých nasledovníkov:

lf = min(ls[succ] for succ in successors)
ls = lf - duration

Piatym krokom je výpočet celkovej rezervy a označenie kritickosti: total_float = ls - es. Úlohy s total_float == 0 sú kritické.

Šiestym krokom je baseline porovnanie — algoritmus sa spustí dvakrát (raz s delay_days, raz bez), čo umožňuje vypočítať celkové oneskorenie projektu spôsobené ručne zadanými oneskoreniami.

def destroy(self):
    """Override — uvoľní matplotlib zdroje pred zničením widgetu."""
    if self._mpl_canvas:
        try:
            self._mpl_canvas.get_tk_widget().destroy()
        except Exception:
            pass
    if self._fig:
        plt.close(self._fig)
    super().destroy()

Automatické doplnenie RETURNING id pre INSERT príkazy je implementované priamo v metóde execute(), takže repozitáre nevyžadujú žiadnu úpravu:

def execute(self, sql: str, params=None):
    pg_sql = sql.replace("?", "%s")
    stripped = pg_sql.strip().upper()
    if stripped.startswith("INSERT") and "RETURNING" not in stripped:
        pg_sql = pg_sql.rstrip(";") + " RETURNING id"
    self._cur.execute(pg_sql, params)
    if self._cur.description:
        self._columns = [d[0] for d in self._cur.description]
        if self._columns == ["id"]:  # uložíme lastrowid
            row = self._cur.fetchone()
            self._last_id = row[0] if row else None

6.2 Dokumentácia chýb a ich opravy

Počas vývoja bolo identifikovaných a opravených 8 signifikantných chýb. Každá bola zdokumentovaná v Obsidian knowledge base s popisom príčiny, dopadom a implementovaným riešením.

Kritická BUG-01: matplotlib canvas konflikt s CTkFrame._canvas

Symptóm: AttributeError: 'FigureCanvasTkAgg' object has no attribute 'bind' pri otvorení záložky Gantt alebo CPM Network.

Príčina: CTkFrame interne používa atribút self._canvas pre vlastné Tkinter plátno. Priradenie self._canvas = FigureCanvasTkAgg(...) prepísalo tento interný atribút, čo spôsobilo pád pri volaní metódy bind() na CTkFrame.

Riešenie: Premenovanie na self._mpl_canvas vo všetkých výskytoch. Nahradenie bind("<Destroy>") za destroy() override, ktorý explicitne uvoľní matplotlib zdroje.

Kritická BUG-02: Slovenské znaky ako čierne štvorce v PDF

Symptóm: Znaky š, č, ž, ľ, á, é, ú sa v exportovanom PDF zobrazujú ako čierne obdĺžniky.

Príčina: Reportlab predvolene používa Helvetica (Type1 PostScript font), ktorá nepodporuje znaky Latin Extended-A (kódová stránka nad U+00FF).

Riešenie: Registrácia TrueType fontu DejaVuSans z matplotlib bundle, ktorý kompletne pokrýva latinku vrátane slovenských diakritík.

Kritická BUG-03: Názov projektu vertikálne orezaný v PDF banneri

Symptóm: Dlhý názov projektu je v PDF banneri orezaný na polovicu výšky.

Príčina: Reportlab Table s VALIGN=MIDDLE a automaticky vypočítanou výškou riadku orezáva text pri väčšom font size.

Riešenie: Reorganizácia bannera na 2-riadkový layout s použitím SPAN a VALIGN=TOP, odstránenie splitLongWords a wordWrap z titulného štýlu.

Kritická BUG-04: color_blend() zlyhá keď BG_CARD je tuple

Symptóm: TypeError pri volaní color_blend(color, bg=BG_CARD) po migrácii na duálny dizajnový systém.

Príčina: Funkcia color_blend() očakávala parameter bg ako jednoduchý hex string, ale po zavedení dual-mode tokenov je BG_CARD tuple (light, dark).

Riešenie: Pridanie parametra dark: bool do signatúry color_blend() a implementácia pomocníka color_blend_dual(hex_tuple, alpha), ktorý generuje blend pre oba módy naraz.

Stredná BUG-05: TclError — pack vs grid geometry manager konflikt

Symptóm: _tkinter.TclError: cannot use geometry manager pack inside ... which already has slaves managed by grid pri otvorení dashboardu.

Príčina: Po refaktoringu pri zavedení scrollovateľného kontajnera bol matplotlib canvas vložený pomocou .pack() do rámca, ktorý pre ostatné widgety používal .grid(). Tkinter nepovoľuje miešanie geometry managerov.

Riešenie: Zmena .pack() na .grid(row=1, column=0, sticky="nsew") s explicitnou konfiguráciou grid_rowconfigure(1, weight=1).

Kritická BUG-06: AttributeError: content_area pri navigácii po páde

Symptóm: Po páde pri inicializácii Dashboardu (BUG-05) každý klik na sidebar generuje AttributeError: '_tkinter.tkapp' object has no attribute 'content_area'.

Príčina: Atribút self.content_area bol inicializovaný až v metóde show_main_layout(), nie v konštruktore __init__. Keď show_main_layout() padlo pred priradením atribútu, sidebar callbacks ho nemohli nájsť.

Riešenie: Pridanie self.content_area = None do __init__ a použitie getattr(self, 'content_area', None) v handle_navigation() ako defensive check.

Stredná BUG-07: KeyError 0 po zavedení Repository Pattern

Symptóm: KeyError: 0 pri otvorení ProjectView, TeamView, ProjectDetail.

Príčina: UI kód pristupoval k dátam cez numerické indexy (user_data[0]) po tom, čo repozitárna vrstva začala vracať slovníky namiesto tuples.

Riešenie: Náhrada všetkých numerických prístupov za pomenované kľúče (user_data["id"]) s defensive pattern pre spätú kompatibilitu: isinstance(data, dict) check.

Nízka BUG-08: border_color="transparent" v CTkFrame

Symptóm: ValueError: transparency not allowed v CalendarView.

Príčina: CustomTkinter akceptuje "transparent" ako fg_color, ale nie ako border_color.

Riešenie: Podmienečné nastavenie border parametrov iba keď je bunka vybraná, pomocou **kwargs rozbalenia slovníka.

Stredná BUG-09: Unit testy — conn.close() zničí :memory: databázu

Symptóm: Unit testy repozitárov zlyhajú s OperationalError: no such table hneď po prvom volaní repozitárnej funkcie.

Príčina: Každý repozitár volá conn.close() v finally bloku. Pre SQLite databázu :memory: toto volanie nenávratne zničí všetky dáta vrátane tabuliek. Ďalší test teda dostane prázdnu databázu bez tabuliek.

Riešenie: Wrapper trieda _NoCloseConn, ktorá potlačí volanie close() a prepustí všetky ostatné metódy cez __getattr__. Zároveň je nutné patchovať každý modul repozitára zvlášť (nie len base_repo.get_connection), pretože Python ukladá importované funkcie ako lokálne referencie v každom module.

7. Záver

7.1 Zhrnutie dosiahnutých výsledkov

Táto bakalárska práca predstavila návrh a implementáciu desktopovej aplikácie ManagmentApp pre projektový manažment s metódou kritickej cesty. Všetky stanovené ciele práce boli úspešne splnené.

Ústredným prínosom práce je implementácia kompletného CPM algoritmu zahŕňajúceho detekciu cyklov (DFS), topologické triedenie (Kahn), dopredný a spätný prechod, výpočet celkovej rezervy a identifikáciu kritickej cesty. Algoritmus bol navrhnutý ako čistá knižnica bez akýchkoľvek GUI alebo databázových závislostí, čo zaručuje jeho ľahkú testovateľnosť a prenositeľnosť.

Ďalším podstatným prínosom je duálny dizajnový systém postavený na natívnych schopnostiach CustomTkinter — dvojice farebných tokenov (light, dark) umožňujú plnohodnotné prepínanie tém bez duplikovania logiky alebo manuálnej aktualizácie jednotlivých widgetov. Tento prístup môže slúžiť ako vzorové riešenie pre iné projekty postavené na tejto technológii.

Architektonické rozhodnutie implementovať Repository Pattern separujúci databázovú logiku od business logiky a UI je investíciou do budúcnosti projektu. Migrácia na cloudové riešenie (PostgreSQL/Supabase) si vyžiada minimálne zmeny — stačí vymeniť implementácie repozitárov bez dotyku na zvyšku kódu.

Aplikácia bola vydaná ako verzia v1.0 MVP s nasledovnými hlavnými funkciami: správa projektov a úloh, plný CPM algoritmus, Gantt diagram, CPM sieťový diagram, správa tímu s trojúrovňovou hierarchiou, kalendárny pohľad, komentáre a prílohy k úlohám, šablóny projektov a export do PDF.

V nadväznosti na MVP bola realizovaná verzia v1.1, ktorá adresovala identifikované technické dlhy: automatizovaná sada 64 unit testov (pytest), live vyhľadávanie úloh, scrollovateľný Gantt chart pre veľké projekty, farebne kódované notifikácie blížiacich sa termínov na dashboarde, rekurzívne BFS zobrazenie celého stromu hierarchie tímu a CSV import/export s validáciou.

Verzia v1.2 zavŕšila pôvodnú architektonickú víziu Repository Pattern: implementáciou duálneho databázového backendu v module base_repo.py je aplikácia transparentne spustiteľná nad lokálnym SQLite aj cloudovým PostgreSQL (Supabase) bez zmeny repozitárov, business logiky ani UI.
